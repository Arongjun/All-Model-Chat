# ADR-0002: 充值订单先做手工确认，再接真实支付网关

## Status
Accepted

## Context

阿荣AI工作站已经具备用户、积分、模型次数包、兑换码、手工调账和使用审计。下一步如果直接接微信、支付宝、Stripe 等支付渠道，容易在业务规则还没稳定时把复杂度提前引进来：订单状态、到账确认、重复回调、退款、人工补单、财务对账都会一起出现。

因此本阶段先落地“充值订单/支付流水雏形”：管理员创建待确认充值订单，用户额度不会立即变化；确认到账后，系统再一次性发放积分和模型次数包，并写入统一审计流水。这样既能覆盖线下收款、企业转账、客服代充值，也能为后续自动支付保留清晰的数据边界。

## Decision

- 新增 `rechargeOrders` 作为订单表雏形，记录订单号、用户、创建人、确认人、金额、币种、支付方式、外部单号、备注、状态和到账额度。
- 订单状态采用最小闭环：`pending`、`paid`、`cancelled`。
- 创建订单只生成待确认记录，不改变用户积分或模型次数包。
- 确认到账时才更新用户额度，并生成 `usageRecords` 事件：`operationType=order_recharge`、`source=order`、`status=success`。
- 取消订单只改变订单状态，不生成入账流水，也不影响用户余额。
- 手工调账继续保留，用于补偿、扣回、纠错；真实充值优先走充值订单，避免财务收入和人工修正混在一起。

## Consequences

### Positive

- 用户可以看到自己的充值订单和到账状态，减少“我付了钱为什么没额度”的沟通成本。
- 管理员能把线下收款转成可审计订单，而不是直接改余额。
- 未来接支付网关时，可以把支付回调映射到 `mark-paid` 语义，而不是重做账务模型。
- 审计导出能区分模型扣费、兑换码、手工调账和订单充值。

### Negative

- 当前还没有支付渠道回调、退款订单、发票/收据和自动对账。
- 单机 SQLite 适合早期运营验证，不适合多实例高并发公开 SaaS 收款。
- 管理员仍需要人工确认到账，自动化程度有限。

## Next

后续接入真实支付时，应优先补齐：

- `paymentTransactions` 或独立支付流水表，记录渠道、回调 ID、原始金额、手续费、渠道状态和幂等键。
- 订单退款/部分退款状态，以及额度扣回策略。
- SQLite 关系表演进，把订单、支付流水、使用事件拆成可索引表；公开 SaaS 阶段再评估 PostgreSQL/MySQL。
- 管理员操作日志，记录谁在什么时候确认/取消了订单。
