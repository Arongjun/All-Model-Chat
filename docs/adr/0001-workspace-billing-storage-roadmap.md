# ADR-0001: 工作站账务先强化审计，再规划数据库迁移

## Status
Superseded by ADR-0003

## Superseded Note

后续实现已按用户要求改为 SQLite 默认存储，不再以 PostgreSQL 作为下一阶段目标。本文保留为账务审计边界的历史决策记录。

## Context
阿荣AI工作站已经从单人聊天工具升级为多用户产品，核心风险从“能不能调用模型”变成“能不能可靠控制成本、解释扣费、处理售后”。当前 MVP 使用 JSON 文件存储，便于本地和轻量部署，但下一阶段需要支持更多成员、兑换码、手工调账、审计导出，以及未来的数据库迁移。

直接把 JSON 替换成 PostgreSQL 可以提升数据可靠性，但如果账务事件模型仍然混在普通使用记录里，迁移后只是把不清晰的数据保存到更强的数据库中。因此需要先把账务事件边界打清楚：模型请求、兑换码、管理员调账都必须形成可查询、可导出的审计记录。

## Decision
当前阶段继续保留 JSON 存储作为默认轻部署方案，同时强化账务审计模型：

- 使用 `usageRecords` 作为统一账务事件流，记录模型请求、兑换、调账。
- 为记录增加 `operationType`，区分 `model_request`、`image_generation`、`redeem`、`admin_adjustment` 等业务类型。
- 为管理员提供分页筛选和 CSV 导出，支持按用户、状态、来源、业务类型和关键词排查。
- 新增手工调账接口，所有积分和次数包的人工增减都必须留下原因和审计记录。
- 将使用记录保留上限从 200 提升到 5000，满足小团队早期运营和排查需要。

未来迁移 PostgreSQL 时，以当前事件流作为表结构基础，而不是直接照搬旧 JSON 文档。

## Consequences

### Positive
- 管理员可以解释“谁在什么时候因为什么扣了多少额度”。
- 线下收款、补偿、误发扣回都有可审计记录。
- 轻部署用户仍然可以直接使用，不强制引入数据库运维成本。
- 后续迁移 PostgreSQL 时，表结构更自然：users、sessions、model_policies、redeem_codes、redemptions、usage_events。

### Negative
- JSON 文件仍不是强并发数据库，不适合大规模公开 SaaS。
- CSV 导出和分页查询目前在内存中过滤，数据量特别大时需要数据库索引支撑。
- 目前还没有租户隔离和订单支付流水，只是为这些能力预留了事件模型。

### Neutral
- 小团队自用或交付给少量客户时，JSON 方案仍然足够轻。
- 如果要对公网开放，必须继续补充 PostgreSQL、支付订单、限流、审计日志分页索引和租户隔离。

## Alternatives Considered

**立即迁移 PostgreSQL**

优点是 ACID、索引和并发更可靠；缺点是当前代码还缺少稳定账务事件边界，贸然迁移会增加部署复杂度，同时没有解决“账为什么这么扣”的产品问题。

**继续只保留余额字段**

实现最快，但管理员无法审计、导出、售后解释，也无法放心做兑换码和线下充值。

**引入完整订单/支付系统**

这是最终方向之一，但当前还没有明确支付渠道和商业规则。先做手工调账和审计导出，能覆盖早期运营场景，也不会锁死后续订单模型。
