# 阿荣AI工作站设计说明

## 目标

在 `All-Model-Chat` 原型基础上，升级为适合真实团队和客户使用的 `阿荣AI工作站`。本次优先落地以下能力：

- 多用户账号体系
- 服务端会话鉴权
- 模型积分与次数包控制
- 兑换码充值
- 管理员后台入口
- 与现有 Gemini / Imagen / GPT 生图链路低侵入集成
- 支持多供应商模型策略，避免系统被单一模型厂商锁死

## 为什么这样设计

原项目的核心优势是前端聊天体验完整，但账号、额度和密钥管理主要偏本地优先。这在单人体验里很好用，但无法真正约束多人使用成本。要解决“给某个用户限制 `gpt-image-*` 可用次数”或“发兑换码给积分”这类需求，控制点必须转移到服务端。

因此本次方案选择：

- 保留原有前端聊天与设置结构
- 把 `/api/gemini` 从“单纯代理”升级为“鉴权 + 配额 + 计费策略 + 代理”
- 在设置面板中新增 `AI 工作站` 页签承载后台能力

这样改造的优点是：

- 不需要推翻原有聊天 UI
- 模型控制真正发生在服务端，不能被前端绕过
- 后续接 OpenAI、Claude 或更多模型时，只需要扩展策略表、密钥配置和代理层

## 架构分层

### 1. 前端层

新增 `WorkspaceSection`，提供：

- 首次初始化管理员
- 登录 / 退出
- 成员兑换码充值
- 管理员用户管理
- 模型计费策略管理
- 兑换码创建与查看
- 最近使用记录

### 2. 服务端能力层

新增 `server/src/workspace.ts`，封装：

- 工作站数据持久化
- 用户密码哈希
- Cookie 会话
- 权限校验
- 模型策略匹配
- 积分与次数包扣减
- 失败回滚
- 兑换码核销

### 3. 代理层

在 `/api/gemini/*`、`/api/openai/*` 请求进入上游前：

1. 识别当前用户
2. 解析模型 ID
3. 匹配模型策略
4. 优先扣次数包，不够再扣积分
5. 生成 pending 使用记录
6. 上游失败时自动退款
7. 上游成功时确认记账

## 数据模型

当前版本使用 SQLite 文件存储，路径默认：

`server/data/arong-workspace.sqlite`

主要实体：

- `users`
  - 用户名、邮箱、角色、密码哈希、积分、模型次数包、禁用状态
- `sessions`
  - 会话 token、用户 ID、创建时间、过期时间
- `modelPolicies`
  - 模型匹配模式、展示名、单次消耗积分、描述
- `redeemCodes`
  - 兑换码、积分、次数包、最大核销次数、过期时间
- `redemptions`
  - 兑换码被谁在什么时候使用
- `usageRecords`
  - 每次预留扣费、成功确认、失败退款的完整轨迹

## 关键流程

### 初始化

- 首次部署时没有用户，系统允许先进入工作站页初始化管理员
- 初始化后自动登录管理员

### 成员使用模型

- 已登录成员发起模型请求
- 服务端根据模型策略进行额度判断
- 若命中次数包，则扣减次数
- 若次数包不足但积分足够，则扣减积分
- 若都不足，则直接拒绝请求

### 兑换码

- 管理员创建兑换码，可附带积分和次数包
- 成员输入兑换码后，系统校验：
  - 是否存在
  - 是否启用
  - 是否过期
  - 是否超过最大使用人数
  - 当前成员是否已兑换过

## 当前实现边界

第一阶段优先做“稳定可运营的 MVP”，因此策略采用：

- 以“每次请求”而非 token 精确计费做额度控制
- 模型策略支持精确匹配和 `*` 前缀匹配
- 数据库存储使用 SQLite 文件，兼顾轻部署、备份和后续拆表演进

第二阶段已补充“运营闭环”：

- 管理员可以手工调账，给用户增加或扣减积分、模型次数包
- 调账必须形成审计记录，保留原因、变化量和操作类型
- 使用记录增加 `operationType`，区分模型请求、生图、兑换和调账
- 管理员可以分页筛选账务记录，并导出 CSV
- 记录保留上限提升，方便小团队早期运营排查

## 下一阶段建议

当你准备把它真正开放给更多人使用时，建议继续做这几件事：

1. 在现有 SQLite 结构化表基础上增加 `payment_transactions`
2. 增加真实支付流水、发票/收据和自动到账
3. 区分文件上传、语音、图片编辑、Live 会话等更细粒度扣费
4. 支持组织 / 租户隔离，而不是单一工作站
5. 接入短信、邮件或企业微信/飞书通知
6. 增加管理员操作日志、异常告警、IP 限流与风控规则
